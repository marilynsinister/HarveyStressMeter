# 🏗️ Архитектура HarveyStressMeter

## 📋 Обзор проекта

**HarveyStressMeter** - мод для Stardew Valley, который добавляет систему стресса с дебаффами и лечением у доктора Харви. Мод использует модульную архитектуру с четким разделением ответственности между сервисами.

## 🎯 Основные компоненты

### 📁 Структура проекта
```
HarveyStressMeter/
├── ModEntry.cs              # Главный класс мода (точка входа)
├── Services/                # Сервисы
│   ├── StateService.cs      # Управление состоянием
│   ├── TreatmentService.cs  # Управление лечением
│   ├── BuffService.cs       # Управление баффами
│   ├── QuestService.cs      # Управление квестами
│   └── TriggerService.cs    # Управление триггерами
├── Models/                  # Модели данных
├── Helpers/                 # Вспомогательные классы
├── Constants/               # Константы
└── UI/                      # Пользовательский интерфейс
```

## 🚀 Жизненный цикл мода

### 1. Инициализация (`Entry()`)
```csharp
public override void Entry(IModHelper helper)
{
    _config = helper.ReadConfig<ModConfig>();           // Загрузка конфигурации
    InitializeServices();                              // Инициализация сервисов
    SubscribeToEvents(helper);                         // Подписка на события SMAPI
    RegisterCommands();                                // Регистрация консольных команд
    LoadTextures(helper);                             // Загрузка текстур
}
```

### 2. Инициализация сервисов
Сервисы создаются в правильном порядке с зависимостями:
- `BuffService` - базовый сервис
- `QuestService` - базовый сервис  
- `StateService` - зависит от BuffService и QuestService
- `TreatmentService` - зависит от всех предыдущих
- `TriggerService` - зависит от всех предыдущих

### 3. Подписка на события SMAPI
```csharp
// Основные события жизненного цикла
helper.Events.GameLoop.SaveLoaded += OnSaveLoaded;      // Загрузка сохранения
helper.Events.GameLoop.DayStarted += OnDayStarted;      // Начало дня
helper.Events.GameLoop.DayEnding += OnDayEnding;        // Конец дня
helper.Events.GameLoop.UpdateTicked += OnUpdateTicked;  // Каждый тик (60 FPS)
helper.Events.GameLoop.GameLaunched += OnGameLaunched;  // Запуск игры
helper.Events.GameLoop.TimeChanged += OnTimeChanged;    // Изменение времени

// События взаимодействия
helper.Events.Display.MenuChanged += OnMenuChanged;     // Изменение меню
helper.Events.Input.ButtonPressed += OnButtonPressed;   // Нажатие кнопок
helper.Events.Player.Warped += OnWarped;                // Перемещение
```

## 🔄 Порядок выполнения игровых событий

### OnGameLaunched (первый запуск)
1. Инициализация UI системы (StardewUI)
2. Регистрация справочника
3. Загрузка текстур

### OnSaveLoaded (загрузка сохранения)
1. **Загрузка данных** - `Helper.Data.ReadSaveData<SaveData>()`
2. **Миграция данных** - `_stateService.MigrateOldData()`
3. **Синхронизация** - `_stateService.SyncWithGame()`
4. **Очистка топиков** - `_treatmentService.CleanupOrphanedTreatmentTopics()`

### OnDayStarted (начало каждого дня)
1. **Синхронизация** - `_treatmentService.SyncQuestsAndBuffs()`
2. **Очистка топиков** - `_treatmentService.CleanupOrphanedTreatmentTopics()`
3. **Восстановление дебаффов** - `_treatmentService.RestoreActiveStressBuffs()`
4. **Проверка триггеров** - `CheckDayStartedStressTriggers()`
5. **Сброс счетчиков** - очистка дневных данных
6. **Сохранение** - `SaveData()`

### OnUpdateTicked (каждую секунду)
1. **Проверка триггеров** - `_triggerService.CheckManualTriggers()`
2. **Обработка тика** - `ProcessGameTick()`
3. **Проверка топиков лечения** - `CheckTreatmentStartTopics()`

### OnDayEnding (конец дня)
1. **Проверка триггеров конца дня** (голод, недосып)
2. **Обновление счетчиков**
3. **Проверка завершения квестов**
4. **Сохранение данных** - `SaveData()`

## 🧩 Архитектура сервисов

### StateService - Центральный сервис состояния
**Назначение**: Единый источник правды для всех операций с состоянием стресса и централизованные проверки.

**Ключевые методы**:
- `ApplyStressBuff()` - применение дебаффа стресса
- `StartTreatment()` - начало лечения (блокировка баффа + квест)
- `CompleteTreatment()` - завершение лечения
- `SyncWithGame()` - синхронизация с игровым состоянием
- `MigrateOldData()` - миграция старых данных

**Централизованные проверки**:
- `HasActiveBuffInGame()` - проверка активного баффа в игре
- `HasQuestInJournal()` - проверка квеста в журнале игры
- `HasActiveTreatment()` - проверка активного лечения
- `HasActiveQuestTreatment()` - проверка активного квеста лечения
- `IsTreatmentLocked()` - проверка блокировки лечения
- `CanIssueNewBuff()` - проверка возможности выдачи нового баффа
- `HasValidTreatmentInHistory()` - проверка валидного лечения в истории

**Управляет**:
- `PlayerStressState.ActiveTreatments` - активные лечения (объединенные баффы и квесты)
- `PlayerStressState.TreatmentHistory` - история всех лечений
- `PlayerStressState.TreatmentFlags` - служебные флаги состояния
- `PlayerStressState.LastIssuedDay` - кулдауны дебаффов

### TreatmentService - Сервис лечения
**Назначение**: Управление процессом лечения стресса.

**Ключевые методы**:
- `ApplyStressBuff()` - простое применение дебаффа (без квеста)
- `StartTreatment()` - запуск программы лечения
- `CompleteTreatment()` - завершение лечения
- `RestoreActiveStressBuffs()` - восстановление активных дебаффов

**Связи**:
- `BuffToQuest` - маппинг дебаффов на квесты
- `BuffToStressTopic` - маппинг дебаффов на топики диалогов
- `BuffToDisplayName` - отображаемые названия дебаффов

### BuffService - Сервис баффов
**Назначение**: Управление баффами/дебаффами в игре.

**Ключевые методы**:
- `ApplyBuff()` - применение кастомного баффа
- `ApplyBuffFromData()` - применение баффа из Data/Buffs
- `RemoveBuff()` - удаление баффа
- `HasBuff()` - проверка наличия баффа

### QuestService - Сервис квестов
**Назначение**: Управление квестами в журнале игрока.

**Ключевые методы**:
- `AddQuest()` - добавление квеста в журнал
- `CompleteQuest()` - завершение квеста
- `HasQuest()` - проверка наличия квеста
- `UpdateQuestDescription()` - обновление описания квеста

### TriggerService - Сервис триггеров
**Назначение**: Проверка условий завершения квестов и обновление прогресса.

**Ключевые методы**:
- `CheckManualTriggers()` - проверка ручных триггеров
- `UpdateTreatmentProgress()` - обновление прогресса лечения
- `UpdateQuestDescription()` - обновление описания квеста

## 🔄 Поток данных

### 1. Применение дебаффа стресса
```
Триггер стресса → TreatmentService.ApplyStressBuff() → StateService.ApplyStressBuff() → BuffService.ApplyBuffFromData()
```

### 2. Начало лечения
```
Согласие в диалоге → TreatmentService.StartTreatment() → StateService.StartTreatment() → QuestService.AddQuest()
```

### 3. Обновление прогресса
```
Каждую секунду → TriggerService.UpdateTreatmentProgress() → StateService.UpdateProgress()
```

### 4. Завершение лечения
```
Условие выполнено → TriggerService.CheckQuestCompletion() → StateService.CompleteTreatment() → QuestService.CompleteQuest() + BuffService.RemoveBuff()
```

## 🎮 Система триггеров стресса

### Триггеры начала дня (OnDayStarted)
- **Tired** - низкая стамина (≤10) при начале дня
- **Thunder** - гроза в любой момент
- **TooCold** - холодная погода в холодных локациях вечером

### Триггеры конца дня (OnDayEnding)
- **Hunger** - отсутствие еды за день
- **NoSleep** - поздний отбой (после 24:00)

### Триггеры времени (OnTimeChanged)
- **Overwork** - обморок от усталости в 2:00 ночи
- **Thunder** - случайный триггер во время грозы

### Триггеры диалогов (OnMenuChanged)
- **Social** - разговор с NPC с низкой дружбой (<3 сердец)

### Триггеры перемещения (OnWarped)
- **Darkness** - темные локации вечером

## 🏥 Система лечения

### Типы дебаффов стресса
1. **Tired** (Усталость) - отдых дома без тяжелых инструментов
2. **Lonely** (Одиночество) - 3 разговора с разными NPC
3. **Thunder** (Страх грозы) - 2 минуты в больнице с Харви во время грозы
4. **Hunger** (Голод) - употребление любой еды
5. **Overwork** (Переработка) - 3 перерыва в зонах отдыха
6. **NoSleep** (Недосып) - 3 дня раннего отбоя подряд
7. **TooCold** (Переохлаждение) - 2 минуты в теплой зоне
8. **Darkness** (Темнота) - 3 минуты вечером в освещенном доме
9. **Social** (Социальный дискомфорт) - 3 разговора + 60 сек с Харви ИЛИ 5 разговоров

### Процесс лечения
1. **Применение дебаффа** - триггер срабатывает → дебафф применяется
2. **Диалог с Харви** - игрок соглашается на лечение
3. **Блокировка дебаффа** - дебафф становится "залоченным"
4. **Выдача квеста** - квест добавляется в журнал
5. **Выполнение условий** - игрок выполняет требования квеста
6. **Завершение** - квест завершается, дебафф снимается

## 📊 Модели данных

### SaveData
Главная модель данных сохранения:
- `PlayerStressState` - состояние системы стресса
- `TalkedNpcsToday` - список NPC, с которыми говорили сегодня
- `OverworkBreaksToday` - количество перерывов сегодня
- `TalkedToHarveyToday` - флаг разговора с Харви

### TreatmentState
Объединенная модель лечения (заменяет StressBuffState + QuestState):
- `BuffId` - ID дебаффа стресса
- `QuestId` - ID квеста лечения
- `IssuedDate` - дата выдачи дебаффа
- `TreatmentStarted` - флаг начала лечения
- `TreatmentStartedDate` - дата начала лечения
- `AddedToGameLog` - флаг добавления в журнал игры
- `IsCompleted` - флаг завершения квеста
- `CompletedDate` - дата завершения
- `IsCured` - флаг излечения
- `Progress` - прогресс лечения (TreatmentProgress)

### TreatmentFlags
Служебные флаги состояния:
- `ActiveTreatmentFlags` - флаги активных лечений
- `QuestAddedToJournalFlags` - флаги добавления квестов в журнал
- Методы для управления флагами и проверки состояния

### PlayerStressState
Центральная модель состояния с объединенной структурой:
- `ActiveTreatments` - активные лечения (объединяет баффы и квесты)
- `TreatmentHistory` - история всех лечений
- `TreatmentFlags` - служебные флаги состояния
- `LastIssuedDay` - даты последней выдачи дебаффов

**Вспомогательные методы**:
- `IsTreatmentLocked()` - проверка блокировки лечения
- `GetLastTreatmentFromHistory()` - получение последнего лечения из истории
- `HasValidTreatmentInHistory()` - проверка валидного лечения в истории
- `GetActiveTreatmentsWithQuests()` - получение активных лечений с квестами
- `GetActiveTreatmentsCount()` - подсчет активных лечений
- `CanIssueNewBuff()` - проверка возможности выдачи нового баффа

## 🔍 Централизация проверок состояния

### Принцип единого источника правды
Все проверки наличия баффов, квестов и лечения проходят через централизованные методы `StateService`:

**Проверки баффов**:
- ❌ `_buffService.HasBuff()` - прямой вызов сервиса
- ✅ `_stateService.HasActiveBuffInGame()` - централизованная проверка

**Проверки квестов**:
- ❌ `_questService.HasQuest()` - прямой вызов сервиса  
- ✅ `_stateService.HasQuestInJournal()` - централизованная проверка

**Проверки лечения**:
- ✅ `_data.StressState.HasActiveBuff()` - проверка активного лечения
- ✅ `_data.StressState.IsTreatmentLocked()` - проверка блокировки лечения
- ✅ `_data.StressState.HasValidTreatmentInHistory()` - проверка валидного лечения в истории

### Архитектура проверок
```
ModEntry/TreatmentService/TriggerService
    ↓
StateService (централизованные проверки)
    ↓
PlayerStressState (состояние данных)
    ↓
BuffService/QuestService (низкоуровневые сервисы)
```

### Преимущества централизации
1. **Единообразие** - все проверки используют одинаковый интерфейс
2. **Надежность** - централизованная логика исключает ошибки
3. **Производительность** - возможность кэширования и оптимизации
4. **Отладка** - легко добавить логирование во все проверки
5. **Тестирование** - простое мокирование централизованных методов

## 🚀 Принципы архитектуры

### SOLID принципы
- **Single Responsibility** - каждый сервис отвечает за одну область
- **Open/Closed** - легко расширяется новыми дебаффами
- **Liskov Substitution** - сервисы взаимозаменяемы
- **Interface Segregation** - четкие интерфейсы
- **Dependency Inversion** - зависимости через конструкторы

### Паттерны проектирования
- **Service Layer** - бизнес-логика в сервисах
- **State Pattern** - управление состоянием через StateService
- **Observer Pattern** - подписка на события SMAPI
- **Dependency Injection** - внедрение зависимостей

### Принципы разработки
- **DRY** - избежание дублирования кода
- **KISS** - простота и понятность
- **Separation of Concerns** - разделение ответственности
- **Single Source of Truth** - StateService как единый источник правды
- **Centralized State Management** - все проверки состояния через централизованные методы
- **Unified Data Model** - объединение связанных состояний в единые модели

## 🔧 Расширяемость

### Добавление нового дебаффа стресса
1. Добавить ID в `Constants/BuffIds.cs`
2. Добавить квест в `Constants/QuestIds.cs`
3. Добавить топик в `Constants/TopicIds.cs`
4. Добавить маппинги в `TreatmentService`
5. Добавить логику в `TriggerService.UpdateSpecificTreatment()`
6. Добавить триггер в соответствующий метод `ModEntry`

### Добавление нового триггера
1. Добавить проверку в соответствующий метод `ModEntry`
2. Вызвать `_treatmentService.ApplyStressBuff()`
3. При необходимости добавить логику в `TriggerService`

- **Эффективные алгоритмы** - оптимизированные методы поиска и фильтрации
